## Root cause (reproduced with the REAL solver + REAL stores, not from screenshots)
Reproduction (5-row 1000 g base, recalc→apply across −13/−11 serving changes): 5 rows/1000 g → 6/1100.6 → 7/1313.0 → 8/1452.4 → 9/2693.7 → **10 rows / 2927.8 g** with `target_batch_grams` still 1000 — Dekstroza ×4 (`l-dex` + `correction-dextrose-0`, `~2`, `~3`), Cream 30% ×2, a parallel milk line. The owner's served ~2937.9 g is this exact accumulation.
- **Where ingredients were appended:** the engine's `applyCorrectionActions` ([verify.ts:67](src/engine/corrections/verify.ts)) pushes a NEW `correction-<ingredient>-<index>` line for every ADD action without `target_line_id` — and the solver sets `target_line_id` ONLY on `reduce` actions ([solver.ts:508](src/engine/corrections/solver.ts)), never on adds. So an add of an ingredient ALREADY in the recipe (same stable `ingredient.id`, e.g. `dextrose`) created a parallel row instead of updating the existing line.
- **Full recipe or delta?** The pipeline contract was already an ATOMIC REPLACEMENT (solver returns a complete next `RecipeInput`; `VerifiedApply.commit` → `structuredClone`; the store swaps `items` wholesale). The defect was INSIDE the proposed input itself — the solver's complete result legitimately contained the appended row, and `ensureUniqueLineIds` ([applyPipeline.ts](src/features/constraint-studio/applyPipeline.ts)) then institutionalized repeats by renaming colliding `correction-*` ids to `~2`/`~3` instead of merging.
- **Why Dextrose/Cream duplicated:** they are both existing recipe lines AND members of the solver's correction toolbox (`dextrose`, `cream_30`, `smp`, `sucrose`, `milk_3_5`…) — every temperature-band fix re-adds them as new rows.
- **Why ~2937.9 g:** each apply increased `sum(planned)` while `target_batch_grams` stayed 1000; nothing rescaled or blocked. Legitimate serving-mode changes between recalcs (−13 ↔ −11) created fresh violations each time, so the additions accumulated across cycles.
- **Double application?** NO — the fingerprint guard already blocks re-applying the same preview (`stale_preview`), and a double-click is a no-op (preview consumed). The accumulation came from SEPARATE legitimate recalcs, each appending.
